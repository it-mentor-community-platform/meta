# End-to-end тесты

End-to-end тесты нацелены на проверку функционала, который может охватывать несколько сервисов или всю системы в целом. Типовой тест кейс начинается с запроса к Gateway, который является триггером к цепочке Kafka взаимодействий, заканчивающейся операциями записи к БД одной или нескольких сервисов. Assert'ы такого тест кейса проверяют итоговое состояние БД.

Пример сложных многоступенчатых взаимодействий, которые хорошо ложатся на данную идею - [учёт проектов](https://github.com/it-mentor-community-platform/meta/blob/main/system-analytics/functionality/projects-bookkeeping.md)

## Стек проекта

E2E тесты - [отдельный проект](https://github.com/it-mentor-community-platform/e2e-tests). Java, Spring Boot 3, Testcontainers, Spring Data JDBC.

### Тестовый контекст

Компоненты:
- Основа - классический `@SpringBootTest`. В рамках контекста, бины для исполнения REST запросов и запросов к БД
- Стек из тестконтейнеров. Всё тоже самое, что в локальном стеке для локального запуска (за исключением фронтенда)

### Структура проекты

- Бины для работы с REST / DB - `src/main/java`
- Тест кейсы, TestContainers - `src/test/java`

### Теги образов для TestContainers

- Запускаем тестконтейнеры из образов с тегом, значение которого передано через переменную `TESTCONTAINER_DOCKER_IMAGES_TAG`. Если значение не установлено - создание стека тестконтейнеров должно кончиться ошибкой
- Есть возможность установить образ конкретному сервису. Например - `AUTH_SERVICE_DOCKER_IMAGE_TAG`. Аналогичные переменные существуют для остальных сервисов. В случае, если переменная не установлена, берём значение из `TESTCONTAINER_DOCKER_IMAGES_TAG`

### CI/CD

- Создание пул реквеста к релизной ветке. Например dev->staging или staging->prod. Стартует пайплайн, запускающий E2E тесты, с `TESTCONTAINER_DOCKER_IMAGES_TAG` выставленной в имя исходной ветки. Данный пайплайн блокирует приём PR
- Создание пул реквеста к из фича ветки к dev. Стартует пайплайн, с `TESTCONTAINER_DOCKER_IMAGES_TAG=dev`, и `*_DOCKER_IMAGE_TAG` конкретного сервиса установленного в имя ветки, из которой создан PR. Данный пайплайн не блокирует приём PR
